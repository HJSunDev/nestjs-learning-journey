# 章节学习引导 (Chapter Learning Guide)

帮助用户系统化学习项目中的章节内容。通过依赖分析生成最优学习路径，逐步引导用户从底层到顶层理解每个文件的设计意图与核心知识点。

## 输入参数

- `$ARGUMENTS`：章节编号（如 `047`）

## 执行流程

### Phase 1: 信息收集与章节概览

**目标**: 收集本章全部素材，为路径规划提供完整信息。

1. 读取 `docs/README.md`，找到对应章节编号的标题与摘要
2. 读取 `docs/notes/{chapter}-*.md` 获取完整设计文档
3. 文件发现（按优先级依次尝试）:
   - **首选**: 从设计文档的"行动导向 (Action Guide)"章节中提取所有 `文件：xxx` 标注的文件路径
   - **补充**: 用 glob 匹配设计文档中提及的目录（如 `src/ai/agents/**/*.ts`）
   - **兜底**: 仅当上述方法不充分时，使用 `git diff --name-only` 作为补充
4. 读取所有发现的文件内容（批量并行读取）
5. 向用户展示章节概览:
   - 章节标题与核心目标（一句话说明本章解决什么问题）
   - 知识点清单（从设计文档的核心概念表中提取）
   - 涉及文件列表（标注新建/修改，并附一句话描述）
   - 预估学习时间（根据文件数量和复杂度估算）

### Phase 2: 依赖分析与路径生成

**目标**: 通过代码依赖关系自动生成学习顺序，而非使用固定模板。

执行以下分析步骤:

1. **构建依赖图**: 分析每个文件的 `import` 语句，提取文件间的依赖关系
2. **拓扑排序**: 从零依赖的叶子节点开始，向上排列到依赖最多的顶层文件
3. **分层分组**: 将文件按依赖层级分组:
   - L0 (基础层): 无项目内依赖的文件（如 State 定义、常量、DTO）
   - L1 (实现层): 依赖 L0 的文件（如节点实现、工具函数）
   - L2 (编排层): 依赖 L1 的文件（如图构建器、Service）
   - L3 (入口层): 依赖 L2 的文件（如 Controller、Module 注册）
   - Lx (聚合层): 纯 re-export 的 index.ts 文件，合并到其目录的主文件步骤中讲解
4. **复杂度标记**: 为每个文件标注学习复杂度:
   - `轻量` — 纯导出/简单配置，1-2 分钟
   - `中等` — 有核心逻辑但较集中，5-8 分钟
   - `重点` — 逻辑密集或架构关键，10-15 分钟
5. **生成学习路径**: 按层级顺序排列，同层内按逻辑关联排序

输出示例格式:

```
### 学习路径

Step 1 [轻量] 设计文档 → 理解核心概念与全景图
Step 2 [中等] agent.state.ts → 理解共享状态定义
Step 3 [中等] call-model.node.ts → 理解 LLM 调用节点
Step 4 [中等] execute-tools.node.ts → 理解工具执行节点
Step 5 [轻量] should-continue.ts → 理解条件路由
Step 6 [中等] tool-graph.builder.ts → 理解图编排
Step 7 [重点] graph.service.ts → 理解 NestJS 桥接层
Step 8 [中等] agent.controller.ts → 理解 HTTP 入口
```

### Phase 3: 逐步引导学习

对每个学习步骤，严格按照以下结构进行讲解:

#### 3.1 上下文铺垫

- 用 1-2 句话说明"接下来要看什么，它在整体架构中的位置"
- 如果有前序章节的对应实现（如 043 的 ToolCallingLoop 对应 047 的 StateGraph），**先展示旧实现的关键代码片段**，说明其局限性，再引出新实现

#### 3.2 代码展示与知识点映射

- 展示文件的核心代码段（不是全部代码，聚焦关键逻辑）
- 将设计文档中的概念**精确映射**到代码位置:
  - 例如: "设计文档中提到的 `MessagesValue` 追加语义，对应这里第 X 行的 `messages: MessagesValue`"
  - 例如: "设计文档中的 `contextSchema 运行时注入`，对应这里第 Y 行的 `config.context as ToolGraphContext`"

#### 3.3 设计决策解析

- 解释"为什么这样设计"而不是"做了什么"
- 如果有取舍（trade-off），说明理由
- 标注设计模式（如纯函数节点、策略路由、桥接模式等）

#### 3.4 关联回顾

- 说明本文件与已学文件的关系（"上一步定义了 State，这一步的节点读取并更新那个 State"）
- 说明本文件如何被后续文件使用（"这个 builder 会在 Service 层被调用"）

#### 3.5 理解检查

提出 1 个聚焦的思考题，帮助确认理解。思考题应该:
- 考察核心概念而非记忆细节
- 有明确的正确答案方向
- 与实际应用场景相关

示例:
- "如果两个并行节点同时更新 toolCallCount，使用普通 Zod 字段（LWW）和 ReducedValue 分别会得到什么结果？"
- "如果把 model 实例放在 State 而不是 contextSchema 中，持久化时会发生什么问题？"

然后等待用户回答或输入 `继续`/`下一步` 跳过。

### Phase 4: 学习总结

完成所有步骤后:

1. **知识图谱**: 以简洁的依赖关系图展示本章所有概念的关联
2. **核心收获**: 3-5 条本章最重要的认知（不是文件清单，而是思维层面的收获）
3. **前序关联**: 本章如何复用/替代前序章节的实现
4. **后续预告**: 下一章将在本章基础上扩展什么能力

## 规则约束

1. **不跳跃**: 严格按照 Phase 2 生成的学习路径顺序进行
2. **不超载**: 每次只讲解一个文件或紧密相关的一组文件（如文件+对应的 index.ts）
3. **互动式**: 每步结束后等待用户确认再继续
4. **聚合文件处理**: 纯 re-export 的 `index.ts` 不单独成步，合并到其主文件步骤中，用一句话带过
5. **概念优先**: 先讲概念再看代码，不要一上来就贴代码
6. **对比驱动**: 只要本章内容有"替代/改进前序实现"的场景，必须展示 before/after 对比
7. **精确映射**: 设计文档中的每个核心概念都必须在代码中找到对应位置
8. **尊重节奏**: 如果用户表示某个概念需要深入讨论，暂停路径推进，先解答清楚
